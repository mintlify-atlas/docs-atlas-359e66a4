## Testing Your Implementation

All example files are documented in `cids.json` with their expected CIDs. To test your implementation:

1. **Generate CIDs** for all example files using your implementation
2. **Compare results** with the expected CIDs in `cids.json`
3. **Verify** that your CIDs match exactly

Each implementation should include:
- `cid.*` - Core library for CID computation
- `generate.*` - Script to generate CIDs for example files
- `check.*` - Script to verify CIDs match expected values

## Using the Online Tools

You can test CID generation without writing code using the online tools:

- **[Hash Calculator](https://256t.org/hash.html)** - Type text and instantly see its SHA-512 hash in Base64URL
- **[CID Examples](https://256t.org/examples.html)** - Select example files and verify their CIDs using the ECMAScript implementation
- **[File CID Check](https://256t.org/check.html)** - Drag and drop files to analyze their content and calculate their CID
- **[CID Download](https://256t.org/download.html)** - Download and verify content from a 256t.org compatible server using a CID

## Key Observations

<Info>
**Inline vs Hashed Storage**

Notice how files â‰¤ 64 bytes have shorter CIDs because the content is stored inline rather than hashed. This is a key optimization in the 256t.org specification.
</Info>

<Tip>
**Deterministic Across Languages**

All implementations produce identical CIDs for the same content, regardless of programming language. This demonstrates the specification's consistency and portability.
</Tip>
